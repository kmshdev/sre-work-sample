Evidence and Validation Guide

Show proof that a reviewer can inspect and rerun

The assignment does not require a production system. It does require evidence that the local slice works from a fresh checkout, fails visibly, and recovers through documented operator steps.

Filter-grade evidence

Acceptable for the filter

  • Fresh-checkout and smoke evidence can be run by the reviewer; captured output supports scenario evidence.
  • The evidence identifies the runtime, state, timestamp, and affected dependency.
  • The failure proof shows trigger, detection, state transition, alert path, and recovery.
  • The report explains what is real, simulated, documented only, or intentionally omitted.

Not enough by itself

  • A green dashboard screenshot with no failure scenario.
  • A health endpoint that only proves the process is alive.
  • An alert name with no threshold, severity, owner, or operator action.
  • AI-generated prose that is not tied to commands, logs, or screenshots.

Minimum evidence set

Area Capture Why it matters
Fresh checkout Exact setup and smoke-test commands with expected output. Reviewers can start from zero without guessing local state.
Smoke test Command output proving the control surface, runtimes, mock dependency, and health response. The slice is locally runnable and exercises the operational model.
Delivery automation Workflow, Make or Taskfile target, CI log, dry-run output, or deployment transcript. Reliability expectations are repeatable rather than manual notes.
Health states Endpoint, CLI, dashboard, or log proof for states exercised by the slice. The health model is connected to observable behavior.
Metrics and logs Metric samples, scrape output, dashboard JSON, or searchable log excerpts. Operators can see impact and diagnose cause.
Alerts Alert rule definitions and examples of page versus ticket severity. The alert route is deliberate, not just noisy monitoring.
Failure injection Two operational failures with trigger, detection, state transition, alert, and recovery proof. Failure handling has been exercised, not only described.
Release safety Bad rollout or bad configuration evidence, or a documented validation and rollback plan. Promotion, rollback, and blast-radius decisions are visible.

Strong versus weak evidence

Strong evidence

  • Includes runtime identity, timestamps, state, and affected dependency.
  • Shows the command or action that triggered the condition.
  • Shows what was blocked or still allowed.
  • Shows recovery and the signal used to close or downgrade.
  • Can be rerun locally without paid services or real secrets.
  • Labels planned-but-not-implemented scenarios separately from implemented evidence.

Weak evidence

  • A screenshot without timestamps, identity, or state context.
  • A log excerpt that cannot be tied to a failure trigger.
  • A dashboard with green panels but no failure case.
  • An alert name without threshold, severity, or operator action.
  • "It restarts" as the only proof of recovery.

Failure scenario checklist

For each implemented failure

  • How to trigger it.
  • How the platform detects it.
  • Which runtime or platform state changes.
  • Which actions are blocked or still allowed.
  • Which alert or incident path fires.
  • How to recover and verify recovery.

Useful commands to include

  • make up, make smoke, or equivalent setup path.
  • Delivery automation command, dry run, or workflow transcript.
  • Health check commands for individual runtimes.
  • Metric scrape or dashboard export command.
  • Failure trigger commands or toggles.
  • Recovery commands and post-recovery checks.
Keep evidence compact. A few targeted excerpts with exact commands are better than pages of logs that reviewers must interpret.

Submission report cross-check

  • Every claim in the report links to a command, screenshot, log excerpt, or file.
  • Every omitted feature has a reason and a production hardening note.
  • AI-generated work is disclosed in AI_USAGE.md if applicable.
  • AI-generated changes are tied to verification you ran yourself.
  • The no-agent incident segment is clearly separated in NO_AGENT_INCIDENT.md.
  • A reviewer can tell what is real, simulated, documented only, or intentionally omitted.
