Health model guide

Running is not the same as safe.

A runtime can answer requests and still be unsafe for work. Your health model must separate process liveness from operational eligibility.

Model shape

Show what the operator can trust.

Process liveness

The runtime is up, reachable, and within basic resource limits.

Operational eligibility

The runtime has fresh data, trusted dependencies, and safe rollout state.

State table

Use only states you can explain with evidence.

State Entry signal Allowed Blocked
healthy Process, data, dependency, and rollout checks are current. Normal work. Nothing beyond normal authorization.
paused_by_operator Operator pauses a runtime or group. Inspect, recover, resume, and audit. Unsafe automated work.
degraded_dependency Mock dependency times out, errors, or returns stale data. Read-only inspection and safe retries. Actions that depend on untrusted dependency state.
maintenance_or_rollout Runtime is in a rollout, canary, or config transition. Promotion checks and rollback. Broad unsafe work until validation passes.

Design questions

Answer these questions in the submission.

  • Which signal moves a runtime into each state?
  • What does the operator see when data is stale or dependency truth is unknown?
  • Which actions fail closed, and which actions remain allowed?
  • How does the runtime recover, resume, or roll back?
  • Which alert, ticket, annotation, or runbook path applies?
Implement evidence for states exercised by your slice. Document other states as future work.
